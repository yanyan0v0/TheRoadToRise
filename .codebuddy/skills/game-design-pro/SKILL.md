# 专业游戏策划案撰写

## Overview

本Skill用于指导AI撰写符合专业游戏策划规范的完整策划案文档。产出为**Markdown + Excel双格式**，内容完全一致，交付不间断完成。撰写风格严格遵循资深系统策划的文风，具备层级分明、逻辑严谨、边界详尽、意图先行的特征。

默认作者信息：**Apakoh(喻骋远)**

## 策划案撰写工作流

按以下步骤顺序执行，不跳步。

### Step 1: 需求分析与类型判断

收到用户的游戏功能/系统/玩法需求后，首先进行分析：

1. **明确需求范围**：确认要设计的是一个完整系统、一个玩法、一条养成线、一个活动，还是一个商业化方案
2. **判断策划案类型**，确定文件命名前缀：

| 类型 | 前缀 | 适用场景 |
|------|------|----------|
| 系统/功能需求 | 【系统】 | 独立功能系统（如氛围气泡、召唤系统） |
| 养成线需求 | 【系统-养成线】 | 角色/装备/遗产等养成线 |
| 常驻玩法需求 | 【常驻玩法】 | 永久开放的玩法（如爬塔、PVP、副本） |
| 活动玩法需求 | 【活动玩法】 | 限时运营活动玩法 |
| 商业化方案 | 【商业化】 | 付费点、商城、定价相关 |

3. **确定文件名**：`{前缀}{功能名称}`，全部使用中文命名
4. **划定设计边界**（⚠️ 核心步骤）：

   a) **列出核心设计范围**：本策划案应覆盖的功能模块有哪些？只包含用户需求直接描述的系统/功能本身的完整逻辑。

   b) **识别关联系统**：本系统在运作中会与哪些外部系统产生交互？这些外部系统**不属于本策划案的设计范围**，只需以接口引用方式提及。

   c) **边界判定原则**：
   - **内部**：本系统自身的数据结构、功能规则、UI界面、配置表、操作流程
   - **外部**：投放本系统产出物的玩法/副本、消耗本系统产出物的其他养成线、本系统的商业化定价方案（除非需求明确要求）
   - **判定标准**：如果某个功能模块可以独立拆分为另一份策划案（有自己的交互/功能说明/配置表），则它属于外部关联系统，不应在本策划案中详细设计

   d) **外部系统引用方式**：对于范围外的关联系统，在功能说明中使用一句话引用即可，格式为：`[具体规则详见【类型】XX系统策划案]`

5. **向用户确认**需求范围、类型判断和设计边界是否正确，如有疑问在此步骤澄清

### Step 2: 撰写Markdown版本

加载 `references/writing-style-guide.md` 和 `references/template-structure.md` 获取详细的写作规范和模板结构。

按以下标准模块顺序撰写Markdown策划案，每个模块对应一个一级标题（`#`）：

#### 标准模块（必写）

1. **# 版本号** — 方案版本号（默认1.0）、设计人（默认Apakoh(喻骋远)）、方案内容、概述（≤50字）
2. **# 交互说明** — 新增美术资产量评估表 + 主要界面流转 + 主要界面详细说明
3. **# 功能说明** — 玩法结构简介 + 分模块功能规则详述
4. **# 红点逻辑** — 各功能入口的红点触发/消除规则（矩阵表格形式）
5. **# 配置需求** — 配置表字段定义（字段名/EN/数据类型/打表类型/字段说明）
6. **# IP&Icon需求** — IP包装需求表 + Icon需求表 + 关卡文案（若有）+ 对话文案（若有）
7. **# 埋点&GM需求** — 新增快照（若有）+ 新增埋点表 + GM命令表

#### 补充模块（按需添加）

- **# 规则QA** — 预期的细节问题补充
- **# 体验规划** — 竞品对标、解锁时机、产出规划（策划内部用）

#### 核心写作规范

- **层级标记**：一级要点用`▸`，二级用`--`，三级用缩进`--`
- **条件判断**：使用`若...则...`、`当...时...`、`仅当...才...`、`一旦...则应...`
- **边界情况**：每个功能必须覆盖掉线/杀进程、资源不足、上限溢出、并发操作、跨天/赛季切换、重复操作等异常处理
- **设计意图**：在复杂规则前说明"由于...因此..."或"核心目的是..."
- **注释引导**：填写指引用`//`前缀标注
- **段落间距**：功能说明中段落之间空两行
- **结构顺序**：先整体概述→分模块详述→每模块内先核心规则→再边界处理
- **设计收束**：严格遵守边界，不泛化到关联系统/玩法/副本

详细规范参考 `references/writing-style-guide.md`。

### Step 3: 生成Excel版本

⚠️ **此步骤紧接Step 2，不间断、不暂停、不等待用户确认**。Markdown版本写完后立即开始Excel版本生成。

加载 `references/excel-format-spec.md` 获取Excel格式规范和JSON Schema定义。

**Excel生成基于【规范】系统策划案模板的真实样式**：脚本复制 `assets/template.xlsx` 作为基底，使用模板中的 theme 颜色、等线字体、填充样式等，确保生成结果与模板视觉风格完全一致。

将Markdown内容结构化为JSON定义：

1. 将Markdown各章节映射为JSON的sheets数组
2. 每个Sheet指定type（version_info/interaction/feature_spec/red_dot/config_spec/ip_icon/tracking_gm/generic）
3. 将Markdown中的层级列表转换为rules结构（level/marker/text/children）
4. 将Markdown中的表格转换为对应的数据数组
5. **确保Markdown和Excel的文本内容完全一致**

然后调用生成脚本：

```bash
python {skill_base_dir}/scripts/generate_design_doc.py <json_file_path> [output_dir]
```

JSON定义的完整Schema参考 `references/excel-format-spec.md`。

**样式说明**（脚本已内置，无需手动设置）：
- Sheet大标题：等线22pt粗体白色 + 深蓝底(theme=4,tint=-0.5)
- 二级标题：等线14pt粗体白色 + 蓝底(theme=4,tint=-0.25)
- 版本号标签：等线12pt粗体白色 + 蓝底
- 版本号值：等线11pt黑色 + 浅底(theme=3,tint=0.9)
- 正文：等线11pt theme=1黑色
- 注释行(//前缀)：等线11pt theme=4蓝色
- 表头：等线11pt粗体白色 + 橄榄绿底(theme=3,tint=0.5)
- 边框：thin样式

#### Sheet类型映射

| Markdown章节 | Sheet名 | JSON type |
|-------------|---------|-----------|
| # 版本号 | 版本号 | version_info |
| # 交互说明 | 交互 | interaction |
| # 功能说明 | 功能说明 | feature_spec |
| # 红点逻辑 | 红点逻辑 | red_dot |
| # 配置需求 | 配置需求 | config_spec |
| # IP&Icon需求 | IP&Icon需求 | ip_icon |
| # 埋点&GM需求 | 埋点&GM需求 | tracking_gm |
| 其他补充模块 | [自定义名] | generic |

### Step 4: 交付与校验

1. 两个版本的文件交付给用户
2. 校验清单：
   - [ ] Markdown和Excel内容完全一致
   - [ ] 所有标准模块完整无遗漏
   - [ ] 文件名符合命名规范（中文+类型前缀）
   - [ ] 设计人默认为Apakoh(喻骋远)
   - [ ] 功能说明覆盖了边界情况
   - [ ] 配置需求表字段定义到可实现粒度
   - [ ] 层级标记系统（▸/--）使用正确
   - [ ] 红点逻辑包含触发和消除两方面
   - [ ] **设计边界检查**：功能说明和配置需求中不包含关联系统的详细设计
   - [ ] **外部引用检查**：关联系统仅以一句话引用方式提及，未展开设计
   - [ ] **内容收束检查**：策划案聚焦于当前方案本身，未泛化到其他系统/玩法

## 交付流程说明

**交付分为两步，但不得间断或暂停**：
1. 先完成Markdown版本策划案文件
2. 紧接着完成Excel版本策划案文件
3. 两个版本的内容保持完全一致

## 各模块内容概览

以下是各标准模块的内容概览，详细结构参考 `references/template-structure.md`。

### 版本号

```
方案版本号: 1.0
设计人: Apakoh(喻骋远)
方案内容: [策划案名称]
概述: [≤50字，格式：通过[手段]，以[目的]。方案核心：[核心设计理念]]
```

### 交互说明

1. **新增美术资产量评估表**

| 分类 | 详细 | 数量 | 说明 |
|------|------|------|------|
| UI界面 | [逐个枚举] | N | [prefab类型] |
| 弹窗 | [特殊单列，通用合并] | N | [prefab类型] |
| Icon | [按类型分组] | N | [说明] |

2. **主要界面流转**：界面间跳转关系和标准操作流程
3. **主要界面详细说明**：复杂界面的UE注解和交互说明

### 功能说明

- 以**玩法结构简介**开篇
- 按功能模块分节，每节内用**▸/--层级标记**描述规则
- 必须覆盖**边界情况**（掉线/杀进程/资源不足/并发/上限/跨天等）
- 设计意图先行阐释
- ⚠️ **严格聚焦**：只写本系统/功能/玩法自身的逻辑，不涉及关联系统的设计

### 红点逻辑

- 使用矩阵表格形式：行=红点情况，列=各UI入口
- ★标记出现红点的入口
- 覆盖触发和消除条件

### 配置需求

- 配置表名称 + 用途说明
- 字段定义表：字段名 | EN | 数据类型 | 打表类型 | 字段说明
- 新增字段标注"*新增"
- 字段说明包含所有枚举值含义
- 表间关联用"索引至"描述

### IP&Icon需求

- IP包装需求表：包装内容 | 包装类型 | 数量 | 文案包装 | 字限建议
- Icon需求表：内容 | 数量 | 规格
- 关卡文案需求（若有）：关卡序号 | 关卡名
- 对话文案规划（若有）：文案 | 语境/触发条件
- 临时名称用"（临时）"前缀标注

### 埋点&GM需求

- 快照需求（若有）：字段名 | 发送时机 | 描述 | 格式 | 归类 | 备注
- 新增埋点表：大类 | 子类 | 归属 | Event | 名称 | 上报字段
- GM命令表：指令 | 参数 | 描述

## 不同策划案类型的特殊要求

### 【系统】系统/功能需求

- 标准模块全套
- 交互说明可能较简单，无复杂交互时可合并至功能说明
- ⚠️ **边界提醒**：只设计本系统自身的功能逻辑和配置，不设计使用本系统的上层玩法或投放场景

### 【系统-养成线】养成线需求

- 标准模块全套
- 配置需求更详细，常含多张表和数值推演
- 可能包含"开发过程中问题记录"补充Sheet
- ⚠️ **边界提醒**：只设计养成线自身的获取/升级/分解等规则，不设计产出这些养成物品的玩法/副本/关卡

### 【常驻玩法】常驻玩法需求

- 标准模块全套
- 交互说明详细，含多界面UE说明
- 功能说明以"玩法结构简介"开篇
- 常包含版本迭代改动Sheet和体验规划Sheet
- ⚠️ **边界提醒**：只设计玩法自身的规则和流程，不设计奖励物品的养成线系统

### 【活动玩法】活动玩法需求

- 标准模块全套
- 重点关注活动周期、开放时间、限时规则
- 商城/兑换和奖励产出设计是重点
- ⚠️ **边界提醒**：只设计活动玩法自身的规则、奖励和兑换逻辑

### 【商业化】商业化方案

- 标准模块全套
- 重点关注付费点设计、定价策略
- 经济系统平衡和转化漏斗
- ⚠️ **边界提醒**：只设计付费点和定价策略，不设计被付费内容关联的系统功能规则
