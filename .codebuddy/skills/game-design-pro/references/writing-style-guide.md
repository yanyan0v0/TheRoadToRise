# 策划案写作风格指南

本文档从多份真实游戏策划案中提取了完整的文风特征与写作规范，作为AI撰写策划案时的核心参考。

---

## 一、层级标记系统

策划案中使用严格的层级标记体系来组织信息，确保逻辑层次清晰：

### 1. 标准层级标记

| 层级 | 标记符号 | 用途 | 示例 |
|------|----------|------|------|
| 一级要点 | `▸` | 核心规则/功能点 | `▸ 副本玩法首先由主线进度控制整个玩法的解锁` |
| 二级说明 | `--` | 对一级要点的补充说明/子规则 | `-- 当玩家正常保存阵容时，同步保存新的奥义牌设置` |
| 三级细节 | 缩进+`--` | 对二级说明的进一步展开 | （在二级--基础上再缩进一格使用--） |

### 2. 层级使用规则

- 一级要点（▸）用于独立的功能规则或条件分支
- 二级说明（--）紧跟在一级要点后方，阐释具体实现细节
- 每条规则独占一行，不要将多条规则合并到一行
- 每个功能点之间空一行分隔
- 相关联的规则组成一个逻辑块，逻辑块之间空两行
- 当内容为场景表现说明（非功能逻辑）时，使用▸标记

### 3. 层级标记示例

```
▸ 副本关卡随主线关卡进度逐步解锁
    -- 当主线进度达到X章节时，解锁对应的副本关卡
    -- 已解锁的关卡不会因主线进度回退而重新锁定

▸ 每日挑战次数限制
    -- 每个副本关卡每日有固定的挑战次数上限
    -- 次数在每日标准游戏时间（05:00）重置
        -- 若玩家在重置时正处于战斗中，则战斗结束后按新一天的次数计算
```

## 二、条件判断句式

策划案中大量使用条件判断来描述规则分支，必须遵循以下句式规范：

### 1. 标准条件句式

| 句式 | 场景 | 示例 |
|------|------|------|
| `若...则...` | 条件触发型规则 | `若玩家当前未拥有此装备，则提示"装备不存在"` |
| `当...时...` | 状态/时机描述 | `当锚点发生移动时，气泡需要随之平滑跟随` |
| `若...否则...` | 二分支判断 | `若已勾选，则自动进入下一关；否则停留在结算界面` |
| `仅当...才...` | 限制条件 | `仅当农民位于摄像机可视范围内时，才计算其气泡生成逻辑` |
| `一旦...则应...` | 即时响应 | `一旦气泡移出屏幕，则应立即销毁并释放配额` |

### 2. 条件判断写法要求

- 条件判断必须穷尽所有分支，不留歧义
- 使用"否则"或"其它情况"明确兜底逻辑
- 复杂条件使用多条▸分别列举，而非一条中嵌套多层
- 涉及数值的条件要给出具体阈值，不使用模糊表述

### 3. 条件判断典型示例

```
▸ 处于装备变更效果的确认界面时，发生掉线/杀死进程等阻断情况，按如下规则处理：
    -- 当前装备的效果变更确认进入挂起状态，等待下一次确认
    -- 在完成确认前，装备词条依然生效为原始效果而非新效果

▸ 战斗开始后一定时间，对敌我双方同时激活全局buff（初始为无buff）
    -- 全局buff配置为BuffID，从skillbuff表读取
    -- （接口需支持，后续线上会视情况拓展）全局buff分等级，每隔一段时间，全局buff提升一级
```

## 三、设计范围约束规范（核心约束）

**这是防止策划案内容膨胀的最关键约束。** 每份策划案只设计"自己该管的事"，对关联系统只做接口级引用。

### 1. "内/外"判定准则

| 判定 | 属于本策划案（内部） | 不属于本策划案（外部） |
|------|---------------------|----------------------|
| 核心特征 | 本系统直接管辖的功能逻辑、数据结构、UI界面、配置表 | 可以独立拆分为另一份策划案的功能模块 |
| 判定方法 | 问自己：这个功能的"需求主人"是本系统吗？ | 问自己：这个功能是否有自己独立的交互/功能说明/配置表？ |
| 举例 | 养成线的获取/升级/强化/分解规则 | 投放养成物品的副本关卡规则 |

### 2. 关联系统引用写法

当功能说明中需要提及外部关联系统时，**只用一句话引用**，不展开设计：

```
✅ 正确写法（引用）：
▸ 圣遗物通过秘境副本掉落获取 [具体副本设计详见【常驻玩法】圣遗物秘境]
▸ 圣遗物可作为养成素材被消耗 [具体消耗规则详见对应养成线策划案]

❌ 错误写法（越界设计）：
▸ 圣遗物秘境分为4个难度等级，每个难度有不同的掉落池...
  -- 简单难度：建议战力2000，掉落3星圣遗物...
  -- 普通难度：建议战力4000，掉落3-4星圣遗物...
  （这些内容属于副本玩法策划案，不属于圣遗物养成线策划案）
```

### 3. 配置表归属规则

配置需求中**只写本系统直接管辖的配置表**：

| 归属 | 处理方式 | 示例 |
|------|----------|------|
| 本系统的表 | 完整列出字段定义 | 圣遗物表（RelicConfig）、圣遗物强化表（RelicUpgrade） |
| 外部系统的表 | 一行引用标注 | `副本掉落表（引用自【常驻玩法】圣遗物秘境，此处不展开）` |
| 需要在外部表新增字段 | 只列出本系统需要的新增字段，标注"*拓展字段" | 在奖励表中新增RelicDropGroup字段 |

### 4. 常见越界场景

以下是常见的设计范围溢出场景，撰写时需要主动规避：

| 策划案类型 | 容易越界到的内容 | 正确做法 |
|-----------|-----------------|---------|
| 养成线需求 | 产出养成物品的副本/关卡设计 | 只写获取途径枚举（如"通过副本掉落"），不设计副本本身 |
| 养成线需求 | 养成物品的商城售卖/定价 | 只写"可通过商城购买"，不设计商城定价 |
| 系统功能需求 | 使用本系统的玩法规则 | 只写本系统提供的接口能力，不设计玩法如何调用 |
| 常驻玩法需求 | 奖励物品的养成规则 | 只写"掉落XX物品"，不设计物品的升级/强化规则 |
| 活动玩法需求 | 奖励道具在常驻系统中的养成逻辑 | 只写奖励列表，不设计道具后续的养成用途 |

### 5. 获取途径的正确写法

当本系统需要说明物品/资源的获取途径时，使用**枚举式引用**，不展开具体规则：

```
▸ 圣遗物获取途径：
    -- 秘境副本掉落 [详见【常驻玩法】圣遗物秘境]
    -- 世界Boss掉落 [详见【常驻玩法】世界Boss]
    -- 商城购买 [详见【商业化】商城方案]
    -- 活动奖励 [由运营活动策划案定义]
```

## 四、边界情况写法

**这是策划案质量的核心体现。** 每个功能设计都必须考虑并详细说明异常/边界情况的处理。

### 1. 必须覆盖的边界场景

| 类型 | 说明 | 处理要求 |
|------|------|----------|
| 掉线/杀进程 | 玩家在操作过程中意外断线或强制关闭客户端 | 明确数据保存/回滚规则 |
| 数据冲突 | 同一数据被多个操作同时修改 | 明确优先级和锁定规则 |
| 资源不足 | 操作所需资源不够时的处理 | 明确提示方式和回退逻辑 |
| 并发操作 | 多个玩家/系统同时操作同一目标 | 明确互斥和等待规则 |
| 上限溢出 | 数量达到上限后的处理 | 明确是否可超限及后续处理 |
| 版本兼容 | 新旧版本客户端交互 | 明确降级和兼容策略 |
| 时间窗口 | 跨时间节点（如跨天、赛季切换）的操作 | 明确归属判定规则 |
| 重复操作 | 玩家快速重复点击或重复提交 | 明确去重/防抖策略和UI屏蔽逻辑 |

### 2. 边界情况写法示例

```
▸ 处于装备变更效果的确认界面时，发生掉线/杀死进程等阻断情况，按如下规则处理：
    -- 当前装备的效果变更确认进入挂起状态，等待下一次确认
    -- 在完成确认前，装备词条依然生效为原始效果而非新效果

▸ 当【同步为进攻阵容】开启时，在阵容保存时，初始牌顺序的信息也需要同步至进攻阵容信息中

▸ 修改防守阵容时，对初始奥义牌设置进行【保存】，会临时存储当前设置。
    -- 当玩家正常保存阵容，并退出编辑模式时，同步保存新的阵容和奥义牌设置
    -- 否则，将阵容和奥义牌设置同时还原至上一版本
```

### 3. 边界情况检查清单

撰写功能说明中的每个核心规则后，对照以下清单逐项检查：

1. 此操作如果在中途被中断（掉线/杀进程），数据会如何处理？
2. 此操作涉及的资源如果不足，会有什么表现？
3. 此操作如果发生在日切/赛季切换等时间节点，归属如何判定？
4. 此操作的前置条件有哪些？前置条件不满足时如何表现？
5. 此操作有无数量/次数/频率上限？达到上限后如何处理？
6. 此操作如果被快速重复触发，是否会导致异常？

## 五、设计意图阐释

策划案不仅要说明"怎么做"，更要说明"为什么这样做"。

### 1. 意图阐释时机

- 在功能说明的开头段落
- 在引入复杂规则前
- 在做出非直觉设计决策时
- 在对标竞品参考时

### 2. 意图阐释句式

| 句式 | 示例 |
|------|------|
| `由于...因此...` | `由于当前玩家队伍坦度逐渐增强，竞技场已经出现大量拖到超时的情况，因此增设全局Buff` |
| `为了...所以...` | `为了保证画面整洁度，所以对同屏气泡数量设置了上限` |
| `核心目的是...` | `方案核心：建立一套有呼吸感的生态系统` |
| `不利于...` | `长期不利于竞技场环境的发展` |

### 3. 概述段落写法

版本号Sheet中的"概述"字段要求在50字以内概括方案的核心设计思路。格式如下：

```
通过增加内城农民的动态气泡反馈，以增强内城模拟经营感与时代沉浸感
方案核心：建立一套有呼吸感的生态系统，通过【背景噪音（较高频率的工作状态指示气泡）】与【高光时刻（NPC文本剧情/心情传达气泡）】的结合，在保证画面整洁度的前提下提供NPC工作信息与氛围感
```

## 六、注释引导格式

策划案中使用 `//` 前缀标注编写指引和策划备注，用于指导策划后续填写或阅读时理解上下文。

### 1. 注释使用场景

| 场景 | 示例 |
|------|------|
| 填写指引 | `//大致枚举一下UI界面、弹窗、Icon、视频、2D动画、Timeline、场景原画、3D资产等潜在美术资产的总量` |
| 格式说明 | `//独立的UI界面要单独拆行枚举` |
| 排除说明 | `//toast提示等不枚举在此列，因为不是新增的美术资产` |
| 示例标注 | `//以下为示例` |
| 生成时移除标注 | `//注释：（生成时请移除本行）` |
| 空行占位 | `//注释：（生成时请移除本行）空2行` |

### 2. 注释写法规则

- 注释行以 `//` 开头，紧跟内容，中间无空格
- 注释文本使用与正文相同的语气（非命令式）
- 注释通常出现在模块开头或示例前方
- 注释仅用于策划沟通/指引，不属于正式功能描述
- **在正式策划案中，以`//注释：（生成时请移除本行）`开头的行必须删除**，这些仅出现在模板中作为填写指引

## 七、版本变更标注

当规则在某个版本中被修改或移除时，使用行内标注记录变更历史。

### 1. 标注格式

```
▸ （24.06.06 已移除此规则）无论玩家选择保留老词条还是替换新词条，在本次变更结束后，清除此装备上所有的【锁定】态
```

### 2. 标注规则

- 使用 `（日期 变更说明）` 格式
- 日期格式为 `YY.MM.DD`
- 变更说明简短明确：`已移除此规则`、`已调整为...`、`新增`
- 标注插入在规则文本最前方，被移除的原文保留但标注为已移除
- 不删除原始规则文本，以便追溯历史

## 八、对标参考写法

在设计时经常需要参考竞品，需要明确标注对标对象。

### 1. 对标表达方式

| 方式 | 示例 |
|------|------|
| 方案名对标 | `遗产系统-对标Nikke魔方` |
| 文内参考 | `Nikke参考：3-4 → Nova规划：第二章末` |
| 效果参考 | `效果参考：钢之炼金术师Mobile - 战争前线玩法` |

### 2. 体验规划表写法

当需要对标竞品时，使用双栏对比表格：

```
| 控制功能 | Nikke参考 | Nova规划 |
|----------|-----------|----------|
| 爬塔玩法入口 | 3-4 | 第二章末 |
| 主塔 | 同上 | 同上 |
| 子塔1 | 5-16 | 第四章中后段 |
```

## 九、数值与公式写法

### 1. 数值表达规则

- 具体数值使用阿拉伯数字：`每日挑战次数上限为5次`
- 范围使用波浪线：`需要消耗10~50个材料`
- 概率使用百分比：`概率100%时必定掉落`
- 公式使用Excel公式格式：`=(N15+N17)/2`
- 万分比标注说明：`万分比，经验值÷银币，若为1:1，则配置为10000`

### 2. 配置字段描述格式

配置字段说明需要详尽到开发可以直接按描述实现：

```
配置塔在已解锁且开启时的每日挑战次数限制

配置阵营ID，代表此塔仅限此ID阵营的角色编队上阵
0=不限制
1=  CampID=1的阵营
2=  CampID=2的阵营
3=  CampID=3的阵营
4=  CampID=4的阵营
```

## 十、功能说明结构模式

功能说明模块遵循固定的结构模式，确保信息完整且有序。

### 1. 标准结构

```
玩法结构简介（概述，含示意图/流程图）

功能说明
    功能模块A标题
        ▸ 规则1
            -- 子规则1.1
            -- 子规则1.2
        ▸ 规则2
            -- 子规则2.1

    功能模块B标题
        ▸ 规则1
        ▸ 规则2
```

### 2. 功能说明写作原则

- **先总后分**：先给出玩法结构简介（含流程图/示意图），再分模块详细描述
- **模块独立**：每个功能模块的规则自成一体，可独立阅读
- **穷尽分支**：每个条件判断必须覆盖所有可能的情况
- **具体可实现**：描述到开发可以直接按文字实现的粒度
- **逻辑优先于表现**：先说功能逻辑，再说UI表现
- **聚焦当前方案**：只描述本系统/功能/玩法自身的功能模块，不展开设计关联系统
- **外部引用格式**：对于关联系统，使用 `[具体规则详见【类型】XX系统策划案]`

## 十一、IP包装需求写法

### 1. 标准表格格式

| 包装内容 | 包装类型 | 数量 | 文案包装（IP填写） | 字限建议 |
|----------|----------|------|-------------------|----------|
| 遗产玩法 | 系统功能名 | 1 | （临时）适格 | ≤4 |
| 玩法内开始战斗 | 行为动词包装 | 1 | （临时）受洗 | ≤2 |

### 2. 包装类型枚举

- **系统功能名**：系统/玩法的主入口名称
- **行为动词包装**：玩家操作行为的包装（如"开始战斗"→"受洗"）
- **玩法子功能名**：系统内子功能/子模块的名称
- **道具名称+描述包装**：道具的名称和详细描述文案
- **关卡名称**：关卡/楼层的名称

### 3. 字限建议规则

- 系统功能名：≤4字
- 行为动词：≤2字
- 子功能名：≤4字
- 道具名：≤7字
- 道具描述：不限
- 关卡名：≤4字

## 十二、埋点与GM需求写法

### 1. 埋点表格式

| 大类 | 子类 | 埋点归属 client/server | Event | 名称 | 上报字段 |
|------|------|----------------------|-------|------|----------|
| 氛围气泡 | 曝光统计 | client | bubble_generate | 生成气泡 | bubble_type;bubble_id |

### 2. 埋点写法规则

- 大类以功能模块命名
- 子类以统计用途命名（如"曝光统计"、"养成"、"系统参与率"）
- 埋点归属明确标注 client 或 server
- Event命名使用下划线连接的英文小写
- 上报字段使用分号分隔
- 如有快照需求，单独列出快照时机和字段

### 3. GM命令表格式

| 指令 | 参数 | 描述 |
|------|------|------|
| gm_show_bubble | targetid;bubbleid | 强制在指定ID的对象身上生成指定ID的气泡 |
| gm_clear_bubble | N/A | 立即销毁所有气泡 |

### 4. GM命令写法规则

- 指令名使用gm_前缀加下划线连接的功能描述
- 参数使用分号分隔，无参数时标注N/A
- 描述要说明指令的完整效果，包括特殊处理

## 十三、红点逻辑写法

### 1. 标准写法

红点逻辑使用矩阵表格形式，行为红点情况描述，列为各UI入口路径：

| 红点情况描述 | 入口A | 入口B | 入口C |
|------------|-------|-------|-------|
| 条件满足时 | ★ | ★ | |
| 另一个条件 | | | ★ |

空单元格代表该入口不受此条件影响。★代表出现红点。

### 2. 红点设计要点

- 明确红点的层级（一级数字红点 vs 二级圆点红点）
- 明确红点的触发入口位置
- 明确红点的消除时机（进入界面/点击/完成操作）
- 明确红点在多层级入口间的传递逻辑

## 十四、交互说明写法

### 1. 美术资产量评估表

交互说明的第一部分是新增美术资产量评估，用表格枚举所有新增的美术资源：

| 分类 | 详细 | 数量 | 说明 |
|------|------|------|------|
| UI界面 | 召唤主界面（不含卡池封面） | 1 | 界面prefab |
| 弹窗 | 卡池说明弹窗*3 | 3 | 弹窗prefab |
| Icon | 系统功能解锁Icon | 1 | 系统解锁弹出（已有） |

### 2. 评估表枚举规则

- 独立的UI界面要**单独拆行枚举**
- 特殊弹窗**单独拆行**枚举；通用弹窗（如说明弹窗）**合并**枚举
- Icon/视频/timeline/原画等按**需求类型枚举**，同类合并，不同类拆行
- toast提示等**不枚举**（非新增美术资产）

### 3. 界面流转与详细说明

- **主要界面流转（概览）**：贴一张所有界面的流转图，说明界面间的跳转关系和标准操作流程
- **主要界面详细说明**：针对复杂界面单独做UE注解图，详细说明界面信息和UX逻辑
- 如有设计连接（arthub、xd、figma），直接贴URL

## 十五、配置需求写法

### 1. 配置表定义格式

```
表名：TribeTower

| 字段名 | EN | 数据类型 | 字段说明 |
|--------|-----|----------|----------|
| TowerID | TowerID | int | 塔的唯一ID |
| 开启日 | OpenDay | string | 配置塔的开启日，格式为1=周一...7=周日 |
| 挑战次数 | Times | int | 配置每日挑战次数限制 |
| 阵营限制 | CampResID | int | 配置阵营ID，0=不限制，1~4=对应阵营 |
```

### 2. 配置需求写法规则

- 每个配置表以表名作为二级标题
- 新增字段使用"*新增"标注
- 字段说明中包含所有枚举值的含义
- 表间关联说明（如"索引至CommonText表"）
- 数据类型使用标准类型：int, string, float, int[], bool, int|m
- 字段说明格式：描述用途 + 枚举值含义（如有）
- 涉及外部表引用的字段说明用"索引至"来描述

### 3. 配置表归属约束

- **只列出本系统直接管辖的配置表**，不列出关联系统的配置表
- 若需在其他系统表中新增字段，只列出新增字段并标注 `*拓展字段`
- 引用其他系统的表时，使用一行引用说明
